Bad practice

Setting local variables inside tap()

A very common pattern is “filter a list and show a result count.” This example filters the list in the stream, but the count is set imperatively in tap() from the unfiltered data.

Why this is a problem

  • The list is filtered in the stream, but the count is set from the unfiltered list.
  • Typing in the filter changes the list, yet the count stays stale.
  • shareReplay does not fix this because the problem is split state.
  • Derived values should come from the stream, not from imperative side effects.

Preferred approach

  • Expose a view-model stream with both list and count.
  • Derive count with map/computed instead of mutating local variables.
  • Keep side effects out of the data pipeline.
